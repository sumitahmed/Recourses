Relational Models in Database

A Relational Model is a data model that organizes data into tables (known as relations) where each table represents an entity or a relationship. In relational models, data is stored in rows and columns, with rows representing individual records (tuples) and columns representing attributes. Each table is identified by a unique name and a primary key that distinguishes each row.

Structure of a Relational Model

In the relational model, data is organized in tables with the following elements:

  • Table (Relation): Each table represents an entity type (e.g., Customer, Order) and contains data about a set of similar items.
  • Row (Tuple): Each row represents a unique instance of an entity in the table, like a single customer or order.
  • Column (Attribute): Each column represents a property of the entity, such as Customer_ID or Name.


Example: A Customer table may have:

Customer_ID Name Contact Email
1 John Doe 555-1234 john@example.com
2 Jane Smith 555-5678 jane@example.com

Domain, Attribute, and Relations

In a relational model, certain key concepts help structure and restrict data within tables:

1. Domain

A domain defines the set of permissible values for a particular attribute, enhancing data consistency.

Example: Domain for Customer_ID: Integer values only.

2. Attribute

An attribute is a characteristic of an entity and is represented as a column in a table. Each attribute has a specific domain that defines the type of values it can hold.

Example: In the Customer table, attributes include Customer_ID, Name, Contact, and Email.

3. Relation

A relation is a table in the relational model that contains data for a specific entity type. Each relation has a unique name and consists of attributes and tuples.

Degree, Cardinality, Relation Instance

1. Degree

The degree of a relation is the number of attributes (columns) it has.

Example: If a Product table has columns Product_ID, Name, and Price, its degree is 3.

2. Cardinality

The cardinality of a relation is the number of rows (tuples) it contains.

Example: If a Product table has 10 rows, the cardinality is 10.

3. Relation Instance

A relation instance is a specific state or set of tuples in a relation at a particular time.

Product_ID Name Price
1 Laptop 1200
2 Smartphone 700

Characteristics of Relations

Relational models possess several characteristics that ensure data integrity and structure consistency:

  • Uniqueness of Rows: Each row in a relation must be unique.
  • Atomic Values: Each attribute value must be atomic (indivisible).
  • Ordering of Rows and Columns: The ordering in relational tables is irrelevant.
  • Primary Key: Each relation has a primary key that uniquely identifies each row.
  • Foreign Key: A foreign key links two relations, establishing a relationship between tables.

Mapping ER Diagrams to Relational Schemas

To build a relational database from an ER model, it’s necessary to map the ER diagram to a relational schema. This process involves converting entities, attributes, and relationships in the ER diagram into tables, columns, and keys in a relational database.

Steps for Mapping ER Diagrams to Relational Schemas

Map Entities to Tables: Each entity in the ER diagram becomes a table.

Map Relationships: Convert relationships in the ER diagram into tables based on their cardinality:

  • One-to-One (1:1): The primary key of one entity can be included as a foreign key in the other entity’s table.
  • One-to-Many (1:M): The primary key of the "one" side is added as a foreign key to the "many" side.
  • Many-to-Many (M:M): A new table is created with foreign keys from both entities.

Example of Mapping: University Database

Consider: A University Database with entities Student, Course, and Instructor with relationships.

Table Columns Description
Student Student_ID (PK), Name, Major Contains student details
Course Course_ID (PK), Title, Credits Contains course details
Instructor Instructor_ID (PK), Name, Department Contains instructor details
Enrollment Student_ID (FK), Course_ID (FK) Links students with courses they enroll in
Teaches Instructor_ID (FK), Course_ID (FK) Links instructors with courses they teach

Difference Between Database and RDBMS

Feature Database RDBMS (Relational Database Management System)
Definition A collection of organized data that can be easily accessed and managed. A type of database that uses tables (relations) to store data in rows and columns.
Data Structure Stores data without a structured format. Stores data in a structured tabular format with relationships.
Relationships Not enforced between data items. Relationships between tables are defined through keys.

Advantages and Disadvantages of RDBMS

Advantages of RDBMS

  • Data Integrity and Consistency: RDBMS enforces rules like primary and foreign keys.
  • Data Security: RDBMS systems provide user authentication and access control.
  • Reduced Data Redundancy: RDBMS reduces data duplication through normalization.
  • Support for Complex Queries: RDBMS allows for complex querying with SQL.
  • Data Independence: Changes to the database schema do not affect the application layer.
  • ACID Compliance: Supports transactions that follow ACID properties.

Disadvantages of RDBMS

  • Complexity and Cost: RDBMS systems can be complex to set up and maintain.
  • Performance Issues: RDBMS may face bottlenecks with large datasets.
  • Fixed Schema Requirement: Less flexible for handling unstructured data.
  • Scaling Challenges: RDBMS is typically vertically scalable, making scaling costly.
