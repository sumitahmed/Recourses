Inference Rules

Inference rules are principles used to derive additional functional dependencies from a given set of functional dependencies. These rules, including Armstrong's Axioms, form the foundation for reasoning about database normalization and consistency.

Real-Life Relation

In a database, if A → B and B → C, inference rules deduce that A → C. This helps validate and simplify schema designs.

Chase and Backchase Algorithms

Chase Algorithm

The Chase Algorithm is used to test properties like dependency preservation and lossless decomposition in database schemas.

Workflow

  1. Start with an initial set of attributes and dependencies.
  2. Iteratively apply dependencies to expand the attribute set.
  3. Check if the target attributes are included.

Real-Life Relation

In schema decomposition, the chase algorithm ensures that no information is lost by verifying that all original dependencies can be derived from the decomposed schema.

Backchase Algorithm

The Backchase Algorithm is used to ensure that constraints are preserved during query rewriting.

Equivalence of Functional Dependencies

Two sets of functional dependencies are equivalent if they produce the same closure, meaning they imply the same set of functional dependencies.

Rule

For sets F1 and F2, F1 and F2 are equivalent if F1+ = F2+.

Real-Life Relation

When optimizing a schema, equivalence helps validate that simplifying functional dependencies does not alter the system's behavior.

Minimal Basis (Canonical Cover)

A minimal basis (or canonical cover) is a simplified set of functional dependencies that is equivalent to the original set. Each dependency in the minimal basis has:

  • A single attribute on the right-hand side.
  • No redundant attributes on the left-hand side.
  • No extraneous dependencies.

Real-Life Relation

In database normalization, a canonical cover ensures that redundancy is minimized while preserving dependency equivalence.

Dependency Preservation

Dependency preservation ensures that all functional dependencies are retained during schema decomposition. It is essential for maintaining data consistency across decomposed schemas.

Rule

A decomposition is dependency-preserving if all functional dependencies can be enforced using only the decomposed relations.

Real-Life Relation

A reIlon database decomposed into Orders and Customers should still preserve dependencies like OrderID → CustomerID.

Lossless Decomposition

A decomposition is lossless if no information is lost when relations are decomposed and then recomposed. Lossless decomposition ensures that the original relation can be reconstructed from its fragments.

Rule

A decomposition is lossless if for relations R1 and R2: R1 ∩ R2 → R1 or R1 ∩ R2 → R2.

Real-Life Relation

In a banking system, decomposing an account table into Account and Branch must be lossless to ensure no account records are lost.
