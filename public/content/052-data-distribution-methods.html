DATA DISTRIBUTION METHODS

Horizontal Fragmentation

Horizontal Fragmentation divides a table into smaller rows or “fragments” based on certain criteria, and each fragment contains a subset of rows. This fragmentation method is commonly used to distribute data based on regional or usage patterns, allowing each fragment to reside closer to the data’s primary users.

Characteristics:

  • Row-based Partitioning: Divides tables by rows, with each fragment containing a subset of rows.
  • Location-based: Data can be fragmented based on geographic or logical criteria.
  • Improved Query Performance: Queries can be optimized by accessing only the relevant fragment.


Example:
  • Fragment 1: Customers in North America
  • Fragment 2: Customers in Europe
  • Fragment 3: Customers in Asia
This fragmentation reduces latency by storing data closer to users in each region.

Vertical Fragmentation

Vertical Fragmentation divides a table by columns, creating fragments with subsets of columns based on query and usage patterns. Each fragment can contain different columns, allowing the database to store frequently accessed columns separately, improving access speed and resource efficiency.

Characteristics:

  • Column-based Partitioning: Splits tables by columns, with each fragment containing specific attributes.
  • Improved Efficiency: Frequently used columns can be stored separately to reduce data retrieval time.
  • Column Dependency: Each fragment must have a key attribute (like a primary key) to enable reconstruction.

Example:

  • Fragment 1: Employee_ID, Name, Department (frequently accessed data)
  • Fragment 2: Employee_ID, Address (infrequently accessed data)

This reduces the load when retrieving frequently accessed columns, optimizing database performance.

Hybrid Fragmentation

Hybrid Fragmentation combines both horizontal and vertical fragmentation, dividing tables into subsets of rows and columns. This approach enables even more granular control over data storage, accommodating diverse query patterns and data distribution requirements.

Characteristics:

  • Flexible Partitioning: Combines the benefits of both row and column-based fragmentation.
  • Optimized for Complex Queries: Suitable for applications with varied query needs.
  • Complex to Manage: Hybrid fragmentation adds complexity to database management and query processing.

Example:

  • Horizontal Fragment: Orders partitioned by region (North America, Europe, Asia).
  • Vertical Fragment: Within each region, columns like Order_ID, Product_ID, and Order_Date in one fragment, and Customer_ID and Shipping_Address in another.

Full vs. Partial Replication

Replication involves storing copies of data across multiple nodes to improve fault tolerance and data availability.

Type Description Example Use Case
Full Replication Stores entire data on all nodes, maximizing fault tolerance and data availability. Financial data where availability is critical
Partial Replication Stores only select fragments on certain nodes, optimizing storage and reducing redundancy. Region-specific data in a multinational company

Full Replication

In Full Replication, every node contains a complete copy of the database. This ensures high availability but increases storage requirements and synchronization complexity.


Partial Replication

In Partial Replication, only certain fragments are stored on each node based on demand or usage patterns, reducing storage needs and synchronizing only critical data.


Centralized Allocation

Centralized Allocation stores all data on a single central server, providing a unified data source for all users. While this simplifies data management, it can create performance bottlenecks and may not be suitable for geographically dispersed applications.

Characteristics:

  • Single Storage Location: All data is located centrally.
  • Easier to Manage: Simplified data consistency and security management.
  • Single Point of Failure: If the central server fails, all users lose access.

Example: A Hospital Database storing all patient data centrally provides doctors and staff with a single source of information. However, this creates a potential bottleneck if the server becomes overloaded.


Distributed Allocation

Distributed Allocation stores data across multiple servers, often geographically distributed to optimize access speed and fault tolerance. This model is well-suited for global organizations needing low-latency access for users across regions.

Characteristics:

  • Multiple Storage Locations: Data is distributed across multiple servers.
  • Improved Fault Tolerance: If one server fails, others continue to operate.
  • Reduced Latency: Data stored closer to users reduces access times.

Example: A Social Media Platform with a distributed allocation stores user data close to their geographic location, improving access speeds and fault tolerance.
