Thomas’ Rules for Timestamp Ordering

The Timestamp Ordering Protocol is essential for concurrency control in databases, ordering transactions by their timestamps to prevent conflicts. Thomas’ Rules extend this protocol, particularly for write operations, to avoid unnecessary transaction aborts and improve system efficiency.

Key Topics

  • Introduction to Thomas’ Rules
  • Limitations of Basic Timestamp Ordering
  • Thomas’ Read Rule
  • Thomas’ Write Rule
  • Timestamp Ordering and Write-Write Conflicts
  • Conflict Handling in Thomas’ Rule

Introduction to Thomas’ Rules

Thomas' Rules are extensions to the basic Timestamp Ordering Protocol and allow certain writes to be skipped or ignored. This prevents unnecessary transaction aborts, especially in write-heavy environments, and improves throughput.

  • Key Principle: Transactions with outdated timestamps may not need to update data if a more recent transaction has already done so. This means we can sometimes ignore certain write operations, reducing conflicts.

Limitations of Basic Timestamp Ordering

In traditional timestamp ordering:

  • Transactions with later timestamps may be unnecessarily aborted due to conflicts.
  • Write-write conflicts often result in newer transactions being rolled back even when their changes wouldn’t affect data integrity.

Thomas’ Rules allow us to bypass these limitations, particularly by ignoring certain outdated writes.

Thomas’ Read Rule

Thomas' Read Rule specifies:

  • A transaction T with timestamp ts(T) can read a data item only if ts(T) is greater than or equal to the item’s Write Timestamp (WTS).
  • If this condition is met, the read proceeds; otherwise, the read is aborted.

Example Table for Thomas’ Read Rule

Step Transaction Operation Condition Action
1 T1 (ts=100) R(A) 100 >= WTS (80) Read Allowed
2 T2 (ts=200) R(A) 200 >= WTS (100) Read Allowed

Thomas’ Write Rule

Thomas' Write Rule allows the system to handle writes more flexibly:

  • Outdated Write Check: If ts(T) < RTS, abort T since the write is outdated.
  • Redundant Write Check: If ts(T) < WTS, ignore T's write as it would be redundant.
  • Proceed with Write: If ts(T) >= RTS and ts(T) > WTS, proceed with the write and update WTS to ts(T).

Timestamp Ordering and Write-Write Conflicts

Timestamp ordering ensures consistency in a database by using transaction timestamps to control the order of operations like read and write. Write-Write conflicts occur when multiple transactions attempt to write to the same data item. These conflicts are managed using Thomas' Write Rule, which allows for outdated or redundant writes to be ignored while ensuring valid writes proceed.

Conflict Type and Resolution

Conflict Type Condition Action
Outdated Write ts(T) < RTS Abort transaction
Redundant Write ts(T) < WTS Ignore write
Valid Write ts(T) ≥ RTS, ts(T) > WTS Proceed with Write

Example of Write-Write Conflict Handling

Assume we have a Customer Balance Table: 

RTS = 200
WTS = 180
Step Transaction Operation Condition Action
1 T1 (ts=150) W(Balance) 150 < RTS (200) Abort T1 (No update)
2 T2 (ts=210) W(Balance) 210 > RTS (200) Proceed with Write (Update WTS = 210)

Explanation:

  • T1 is aborted as it would disrupt consistency.
  • T2 successfully writes, updating the WTS to 210.

Scenario Setup

Consider an Inventory Management System with a table Product that stores information about the stock level of products: 

CREATE TABLE Product (
    ProductID INT PRIMARY KEY,
    ProductName VARCHAR(50),
    StockLevel INT,
    RTS INT DEFAULT 0,     -- Read Timestamp
    WTS INT DEFAULT 0      -- Write Timestamp
);

Table Fields:

  • RTS (Read Timestamp): The last timestamp when StockLevel was read.
  • WTS (Write Timestamp): The last timestamp when StockLevel was updated.

Let’s assume two transactions:

  • Transaction T1 with a timestamp ts(T1) = 100
  • Transaction T2 with a timestamp ts(T2) = 200

Initial Table State

ProductID ProductName StockLevel RTS WTS
1 Widget 50 80 90

Condition Check for Read

According to Thomas’ Read Rule, a transaction can read a data item if ts(T) ≥ WTS. 

-- T1 attempts to read
SELECT CASE
    WHEN 100 >= WTS THEN 'Read Allowed'
    ELSE 'Abort Read'
END AS T1_Read_Status
FROM Product
WHERE ProductID = 1;

-- T2 attempts to read
SELECT CASE
    WHEN 200 >= WTS THEN 'Read Allowed'
    ELSE 'Abort Read'
END AS T2_Read_Status
FROM Product
WHERE ProductID = 1;

Output:

  • T1: Since 100 ≥ 90, the read is allowed.
  • T2: Since 200 ≥ 90, the read is also allowed.

After each read, we should update the RTS to the timestamp of the transaction:

-- Update RTS after successful read
UPDATE Product
SET RTS = 100  -- For T1's read
WHERE ProductID = 1;

-- Updating RTS again after T2's read
UPDATE Product
SET RTS = 200
WHERE ProductID = 1;

Table State After Read

ProductID ProductName StockLevel RTS WTS
1 Widget 50 200 90

Thomas’ Write Rule Example with SQL

Scenario

Transaction T1 (timestamp 100) and Transaction T2 (timestamp 200) both want to update the StockLevel for ProductID = 1.

Initial Table State

ProductID ProductName StockLevel RTS WTS
1 Widget 50 200 90

Conditions for Write

Using Thomas’ Write Rule:

  • Abort T1 if ts(T1) < RTS.
  • Ignore T1 if ts(T1) < WTS.
  • Proceed with the write if ts(T1) ≥ RTS and ts(T1) > WTS.

SQL Example

Step 1: T1 attempts to write
-- T1 attempts to write to StockLevel
SELECT CASE
    WHEN 100 < RTS THEN 'Abort T1 (Outdated Write)'
    WHEN 100 < WTS THEN 'Ignore T1 Write (Redundant Write)'
    ELSE 'Proceed with T1 Write'
END AS T1_Write_Status
FROM Product
WHERE ProductID = 1;

Output for T1: Since 100 < RTS (200), T1’s write is aborted due to an outdated timestamp.

Step 2: T2 attempts to write
-- T2 attempts to write to StockLevel
SELECT CASE
    WHEN 200 < RTS THEN 'Abort T2 (Outdated Write)'
    WHEN 200 < WTS THEN 'Ignore T2 Write (Redundant Write)'
    ELSE 'Proceed with T2 Write'
END AS T2_Write_Status
FROM Product
WHERE ProductID = 1;

Output for T2: Since 200 ≥ RTS (200) and 200 > WTS (90), T2’s write is allowed.

Updating the Table

Since T2’s write is allowed, we update the StockLevel and set the WTS to 200. 

-- T2's successful write to StockLevel
UPDATE Product
SET StockLevel = 60,  -- New stock level updated by T2
    WTS = 200
WHERE ProductID = 1;

Final Table State

ProductID ProductName StockLevel RTS WTS
1 Widget 60 200 200

Example with Write-Write Conflict Handling

In real-world scenarios, write-write conflicts are common. Let’s go through an example where two transactions attempt to write to the same data item and we apply Thomas' Rules to resolve these conflicts.

Scenario

Transaction T3 with timestamp 300 attempts to write StockLevel. Transaction T4 with timestamp 150 also attempts to write StockLevel.

Initial Table State for Write-Write Conflict

ProductID ProductName StockLevel RTS WTS
1 Widget 60 200 200

SQL Example for Write-Write Conflict

Step 1: T3 attempts to write
-- T3 attempts to write to StockLevel
SELECT CASE
    WHEN 300 < RTS THEN 'Abort T3 (Outdated Write)'
    WHEN 300 < WTS THEN 'Ignore T3 Write (Redundant Write)'
    ELSE 'Proceed with T3 Write'
END AS T3_Write_Status
FROM Product
WHERE ProductID = 1;

Output for T3: Since 300 ≥ RTS and 300 > WTS, T3 is allowed to write.

Step 2: T4 attempts to write
-- T4 attempts to write to StockLevel
SELECT CASE
    WHEN 150 < RTS THEN 'Abort T4 (Outdated Write)'
    WHEN 150 < WTS THEN 'Ignore T4 Write (Redundant Write)'
    ELSE 'Proceed with T4 Write'
END AS T4_Write_Status
FROM Product
WHERE ProductID = 1;

Output for T4: Since 150 < RTS (200), T4’s write is aborted due to an outdated timestamp.

Final Table Update after T3's Write

Since T3 was allowed to write, we update StockLevel and set WTS to 300. 

-- T3's successful write to StockLevel
UPDATE Product
SET StockLevel = 70,  -- Updated stock level by T3
    WTS = 300
WHERE ProductID = 1;

Final Table State after Conflict Resolution

ProductID ProductName StockLevel RTS WTS
1 Widget 70 200 300
