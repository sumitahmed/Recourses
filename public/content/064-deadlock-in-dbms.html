Deadlocks in DBMS

A deadlock is a situation in a Database Management System (DBMS) where two or more transactions are waiting indefinitely for resources locked by each other. This situation creates a cycle of dependencies, preventing any transaction involved from proceeding, which can lead to a severe system stall.

Key Characteristics of a Deadlock:

  • Circular Waiting: Transactions are locked in a cycle, each waiting on resources held by another.
  • Resource Contention: Limited resources are occupied by different transactions, causing them to be unable to proceed.
  • Indefinite Blocking: Transactions cannot release resources or proceed, resulting in a system bottleneck.

Real-Life Analogy:

Imagine two cars (T1 and T2) approaching a narrow bridge from opposite ends. The bridge is too narrow for both to pass at the same time, and each car refuses to reverse. Both T1 and T2 are waiting indefinitely, creating a deadlock situation where neither can proceed until one yields.

Deadlock Scenarios in DBMS

Deadlocks can occur in different types of locking situations in DBMS, such as Read-Read (R-R) Locking, Read-Write (R-W) Locking, and Write-Write (W-W) Locking. Each scenario has unique characteristics and outcomes.

1. Read-Read (R-R) Locking

Scenario: Multiple transactions hold shared (read) locks on the same resource simultaneously.

Outcome: Since read locks are non-exclusive, multiple transactions can read the data without interfering with each other. No deadlock occurs in this situation.

Example: In an e-commerce system, multiple customers can view a product’s details (read access) without any conflict.


2. Read-Write (R-W) Locking

Scenario: One transaction holds a read lock, while another transaction requires a write lock on the same resource.

Outcome: The write transaction must wait until the read transaction releases the lock, potentially causing a delay.

Example: In a reservation system, if T1 is reading seat availability while T2 wants to update the seat status, T2 must wait for T1 to complete its read operation.


3. Write-Write (W-W) Locking

Scenario: Two transactions attempt to acquire exclusive (write) locks on the same resource simultaneously.

Outcome: Only one transaction can hold the write lock at any given time, causing the other transaction to wait indefinitely.

Example: In a banking system, if T1 is updating Account_A and T2 also attempts to update it, T2 must wait until T1 completes its operation.


Wait-For Graphs (WFG) in Deadlock Detection

A Wait-For Graph (WFG) is a directed graph used to detect deadlocks. Each node in the graph represents a transaction, and each directed edge represents a wait condition (i.e., T1 → T2 means T1 is waiting for T2 to release a resource).

How to Use a Wait-For Graph:

  • Nodes: Each transaction in the system is represented as a node.
  • Edges: An edge from one node to another indicates that the first transaction is waiting for a resource held by the second transaction.
  • Cycle Detection: A deadlock is detected if there is a cycle in the WFG.

Techniques to Prevent Deadlocks

Deadlocks can be prevented by managing how resources are allocated to transactions. Here are some effective techniques for deadlock prevention:

Condition Real-Life Analogy Technique Description
Mutual Exclusion Exclusive access to a resource, like a single-use bathroom, where only one person can use it at a time. Use a Locking Mechanism Ensure that resources are shared when feasible, avoiding exclusive access unless absolutely necessary.
Hold and Wait A student holds a library book while waiting for another book on hold, preventing others from accessing the first book. Resource Request Protocol Request all resources at once or release held resources before requesting new ones.
No Preemption A person reserves a parking space and won’t give it up, even if others are waiting. Preemptive Resource Allocation Prevents deadlock by permitting the DBMS to abort transactions and release held resources if needed.
Circular Wait Cars at a four-way stop all wait for each other, causing gridlock. Ordering Resources Reduce circular dependencies by enforcing a strict order for resource requests.

Detailed Explanation of Techniques:

1. Mutual Exclusion

  • Scenario: Only one transaction can access a resource, making exclusive access mandatory.
  • Solution: Avoid exclusive locks wherever possible and allow shared access.
  • Real-Life Example: In a coworking space, individuals can share tables instead of occupying them exclusively, allowing more users to access the workspace.

2. Hold and Wait

  • Scenario: A transaction holds resources while waiting for additional resources, increasing deadlock risk.
  • Solution: Require transactions to request all necessary resources at the beginning or release held resources before requesting new ones.
  • Real-Life Example: In a car rental, a customer must specify all equipment they need upfront (e.g., GPS, car seat) to avoid delays.

3. No Preemption

  • Scenario: Resources held by a transaction can’t be forcibly taken, even if they’re needed by others.
  • Solution: Implement preemption, allowing the system to release resources by rolling back transactions.
  • Real-Life Example: In a hotel, if a guest overstays, the hotel may ask them to leave (roll back) so other guests can use the room.

4. Circular Wait

  • Scenario: Transactions are waiting on each other in a circular dependency.
  • Solution: Enforce a strict order for resource requests, breaking circular dependencies.
  • Real-Life Example: At a four-way stop, assigning right-of-way rules allows cars to proceed without getting into a deadlock.

Example Scenario Applying Techniques to Prevent Deadlocks

Consider a Manufacturing System where three transactions (T1, T2, T3) require resources (Machines, Materials, Labor):

  • T1 requests Machine, waits for Materials.
  • T2 requests Materials, waits for Labor.
  • T3 requests Labor, waits for the Machine.

Using the Ordering Resources technique, we can prevent deadlock by requiring all transactions to request resources in a fixed order (e.g., Machines → Materials → Labor).

By following this order, circular dependencies are eliminated, as no transaction can wait indefinitely for a resource held by another transaction.
