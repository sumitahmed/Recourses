Concurrency Control in Distributed Databases

Concurrency control ensures that multiple transactions can access a distributed database simultaneously without causing data inconsistencies or integrity issues. In a distributed system, transactions may occur across different networked sites, making concurrency control even more crucial to prevent conflicts and ensure data consistency.

Real-Life Analogy:

Imagine a multi-branch library system where readers across different locations can access the same digital catalog. Each branch allows users to borrow, reserve, or return books concurrently. Without proper control, actions like double-booking or incorrect stock counts could occur, leading to confusion. Concurrency control in distributed databases works similarly, ensuring that each "branch" or transaction operates without interfering with others.

Techniques for Concurrency Control

1. Two-Phase Locking (2PL)

Two-Phase Locking (2PL) is a widely used concurrency control protocol that ensures serializability by dividing a transaction’s life into two phases:

  • Growing Phase: The transaction acquires all the locks it needs but cannot release any locks.
  • Shrinking Phase: Once it releases a lock, it cannot acquire any new ones.

Example: Applying 2PL in a Distributed Database

Consider a distributed Banking System where:

  • Transaction T1 at Site A wants to transfer money and lock Account 1 and Account 2.
  • Transaction T2 at Site B also wants to lock Account 2 and Account 3 for processing.

Process:

  • Growing Phase: T1 locks Account 1 and Account 2. T2 waits because Account 2 is locked by T1.
  • Shrinking Phase: Once T1 finishes and releases the locks, T2 can acquire them.

2. Timestamp Ordering

Timestamp Ordering is a concurrency control method that assigns a unique timestamp to each transaction when it begins. The database uses these timestamps to determine the order in which transactions should occur, avoiding conflicts by following a strict timestamp-based sequence.

How Timestamp Ordering Works:

  • Transaction Start: Each transaction receives a unique timestamp, determining its order.
  • Operation Check: When a transaction attempts to read or write data, its timestamp is compared to the data’s Read Timestamp (RTS) and Write Timestamp (WTS).
  • Conflict Resolution:
    • Read Rule: A transaction can read data only if its timestamp is greater than the WTS.
    • Write Rule: A transaction can write only if its timestamp is greater than both RTS and WTS.

Example: Timestamp Ordering in an Inventory System

Consider an Inventory Management System where:

  • T1 (older, timestamp 100) tries to read and write Product Stock.
  • T2 (newer, timestamp 200) tries to write to the same Product Stock.


3. Multi-Version Concurrency Control (MVCC)

Multi-Version Concurrency Control (MVCC) allows multiple versions of a data item to exist, enabling transactions to read from different versions without blocking each other. MVCC is commonly used in systems like PostgreSQL and Oracle.

How MVCC Works:

  • Version Creation: Each transaction creates a new version of a data item instead of overwriting it.
  • Version Access: Transactions can read the version that was committed before their start time.
  • Garbage Collection: Old versions are removed once they are no longer needed.

Real-Life Example of MVCC:

In a Hotel Booking System, multiple customers may be reading room availability at the same time while others are booking. MVCC creates versions of the room data:

  • Each reader sees a consistent snapshot without being blocked by other transactions.
  • Writers create new versions, ensuring other transactions can continue without waiting.

4. Validation-Based Concurrency Control

Validation-Based Concurrency Control uses a three-phase approach to validate transactions before committing, ensuring they do not conflict with other concurrent transactions. It is suitable for systems with low contention, where conflicts are infrequent.

Steps in Validation-Based Concurrency Control:

  • Read Phase: The transaction reads all the data it needs without making any changes to the database.
  • Validation Phase: The transaction is validated to check if it conflicts with any other transaction.
  • Write Phase: If validation is successful, the transaction writes changes to the database.

Example: Validation-Based Control in an E-commerce System

In an E-commerce System:

  • T1 and T2 both attempt to purchase the last available item.
  • During validation, the system checks if their operations conflict.
  • Only one transaction succeeds in reserving the item; the other is rolled back.
