Serializability in DBMS

Serializability is the highest isolation level in DBMS, where the outcome of executing concurrent transactions is the same as if they were executed in a strictly serial order. This level of isolation prevents any form of data inconsistency and provides maximum data integrity.

Real-Life Analogy:

Consider a hospital pharmacy where each pharmacist (transaction) is assigned one prescription at a time, ensuring that each prescription is processed without any overlap. This prevents any chance of medication errors or duplications, similar to how serializability prevents conflicts between transactions.

Example: Serializable Schedule

T1 T2
read (X)
write (X)
read (Y)
write (Y)
read (X)
write (X)
read (Y)
write (Y)

Phantom Reads

A Phantom Read occurs when a transaction reads a set of rows that satisfy a certain condition, but another transaction inserts, updates, or deletes rows in the dataset, causing the first transaction to see a different set of rows if it re-executes the same read query.

Example: Phantom Read Scenario

In a Library Database:

  • T1: Reads all books authored by "Author X."
  • T2: Inserts a new book by "Author X."

If T1 reads again, it will see an additional "phantom" row, causing inconsistency.

Example in Table Format

T1 T2
read (Books by Author X)
insert (New Book by X)
read (Books by Author X) (Sees a new row)
This table shows that T1’s second read will display an additional row if T2 inserts a new book in between, creating a phantom read effect.

Non-Repeatable Reads

A Non-Repeatable Read occurs when a transaction reads a data item twice and sees different values because another transaction modified the data item in between the two reads. This inconsistency can occur in Read Committed and Repeatable Read isolation levels.

Example: Non-Repeatable Read Scenario

In a Banking System:

  • T1: Reads the balance of Account A.
  • T2: Updates the balance of Account A.
  • When T1 reads Account A’s balance again, it sees a different value.

Example in Table Format

T1 T2
read (Account A Balance)
write (New Balance of A)
read (Account A Balance) (Sees a different balance)

Impact of Isolation Levels on Concurrency

The choice of isolation level affects the concurrency of transactions in a DBMS. Higher isolation levels (like Serializable) reduce concurrency because transactions have to wait longer to access locked data, while lower levels (like Read Uncommitted) improve concurrency but increase the risk of inconsistencies.

Concurrency Impact at Different Isolation Levels

  • Serializable: Lowest concurrency but highest consistency.
  • Repeatable Read: Prevents dirty and non-repeatable reads; good balance.
  • Read Committed: Allows concurrency but has non-repeatable reads.
  • Read Uncommitted: Maximum concurrency, high risk of inconsistencies.

Choosing the Right Isolation Level

Choosing an appropriate isolation level depends on the requirements for consistency and acceptable risk for inconsistencies in a given application:

  • Banking Systems: Require Serializable for consistency in transactions.
  • E-commerce Systems: Can use Repeatable Read to balance consistency and performance.
  • Analytics Platforms: Often use Read Committed or Read Uncommitted for fast data access.
