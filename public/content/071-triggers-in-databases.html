What are Triggers in Databases?

A trigger is a stored procedure in a database that is automatically executed, or "triggered," in response to specific events on a particular table or view. Triggers help enforce business rules, maintain data integrity, and automate database tasks.

Real-Life Analogy:

Imagine a smart home system that automatically turns on the lights when it detects someone entering a room. The action (turning on lights) is "triggered" by the event (detecting someone). Similarly, in databases, triggers perform specific actions when certain events occur, such as inserting or updating data.

Types of Triggers

There are three main types of triggers based on when they execute relative to the event: Before Triggers, After Triggers, and Instead of Triggers.

1. Before Triggers

Before Triggers are executed before the triggering event, such as an INSERT, UPDATE, or DELETE operation, is processed on the table. They are often used for data validation or modification before the data is saved to the database.

Example: Before Insert Trigger

In a Payroll System:

  • Trigger: Before inserting a new salary record, check that the salary is within a predefined range.
  • Action: If the salary is out of range, prevent the insert operation.

2. After Triggers

After Triggers are executed after the triggering event has occurred. They are typically used for auditing, logging, or additional calculations that depend on the final state of the data.

Example: After Insert Trigger

In an Inventory Management System:

  • Trigger: After inserting a new product record, update the inventory log.
  • Action: Record the addition of the new product in an audit log.

3. Instead of Triggers

Instead of Triggers are used mainly with views to replace the default action of the triggering event. When an INSTEAD OF trigger is defined, the original operation (like INSERT, UPDATE, or DELETE) is skipped, and the trigger’s logic is executed instead.

Example: Instead of Update Trigger

In a View on Sales Data:

  • Trigger: Instead of updating the view directly, redirect the update to the underlying tables.
  • Action: Update related tables as per business rules.

Row-Level vs. Statement-Level Triggers

Triggers can be classified based on the level at which they operate:

1. Row-Level Triggers

Row-Level Triggers execute once for each row affected by the triggering event. They are useful when specific actions need to be taken for each row in a batch operation.

Example: Row-Level Trigger

In a Banking System:

  • Trigger: After updating an account balance, log the change.
  • Action: Each account in a batch update will have a separate log entry.

2. Statement-Level Triggers

Statement-Level Triggers execute only once for the entire SQL statement, regardless of how many rows are affected. They are ideal for logging or enforcing constraints that apply at the statement level.

Example: Statement-Level Trigger

In a User Registration System:

  • Trigger: After inserting multiple users in a batch, log the action.
  • Action: Create a single log entry for the entire batch insert.

Trigger Events

Trigger Events are the actions that activate a trigger. Common trigger events include:

  • INSERT: Trigger fires when new data is added to a table.
  • UPDATE: Trigger fires when existing data in a table is modified.
  • DELETE: Trigger fires when data is removed from a table.

Each trigger event can be associated with Before, After, or Instead of actions to control when the trigger fires relative to the event.

Enabling, Disabling, and Dropping Triggers

Enabling and Disabling Triggers

Triggers can be enabled or disabled to control whether they execute in response to events. Disabling triggers is useful during data migration or bulk operations to prevent unwanted triggers from firing.

Example Commands:

    -- Enable a trigger
    ALTER TRIGGER trigger_name ENABLE;

    -- Disable a trigger
    ALTER TRIGGER trigger_name DISABLE;

Dropping Triggers

If a trigger is no longer needed, it can be permanently dropped from the database.

Example Command:

    -- Drop a trigger
    DROP TRIGGER trigger_name;

Detailed Examples for Triggers

Example 1: Before Insert Trigger to Validate Data

Consider an Employee Database where salaries must be validated:

    CREATE TRIGGER validate_salary
    BEFORE INSERT ON employees
    FOR EACH ROW
    BEGIN
    IF :NEW.salary < 0 THEN
        RAISE_APPLICATION_ERROR(-20001, 'Salary cannot be negative');
    END IF;
    END;

Outcome: If an attempt is made to insert a record with a negative salary, the trigger will raise an error, and the insert operation will be prevented.

Example 2: After Insert Trigger to Log Changes

In an Inventory System, log each addition to the products table:

    CREATE TRIGGER log_new_product
    AFTER INSERT ON products
    FOR EACH ROW
    BEGIN
      INSERT INTO product_log (product_id, action)
      VALUES (:NEW.product_id, 'New product added');
    END;

Outcome: When a new product is added to the products table, an entry is automatically added to the product_log table.
